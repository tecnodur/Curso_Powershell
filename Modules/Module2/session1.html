1. Scripts

O que é um script PowerShell?

Um script PowerShell é um conjunto de comandos guardados num ficheiro de texto com a extensão .ps1. Ao contrário de comandos que são executados diretamente na linha de comandos (modo interativo), os scripts permitem guardar lógica mais complexa, reutilizável e organizada para ser executada de forma automatizada.

1.1. Comandos interativos vs scripts guardados

Tipo Exemplo Descrição
Interativo Get-Process Comando único executado diretamente no terminal. Ideal para testes rápidos.
Script (.ps1) meu_script.ps1 Contém múltiplos comandos, lógica de decisão, loops e funções. Usado para automatizar tarefas.

1.2. Vantagens de usar scripts

  • Guardar e reutilizar lógica complexa
  • Automatizar tarefas repetitivas
  • Documentar o que foi feito
  • Facilitar a partilha de soluções com outras pessoas

1.3. Onde criar scripts

Podes criar scripts com qualquer editor de texto, mas os mais recomendados são:

  • Visual Studio Code com a extensão PowerShell
  • PowerShell ISE (pré-instalado em muitas versões do Windows)
  • Notepad (menos recomendado, pois não tem destaques de sintaxe)

1.4. Criar o primeiro script

Os scripts PowerShell devem começar com comentários (#) que ajudam a documentar o que o script faz. A seguir, colocas os comandos que queres executar.

Exemplo de script simples (ola.ps1)

# ola.ps1
# Este script imprime uma saudação

$nome = "Nuno"
Write-Output "Olá, $nome! Bem-vindo ao PowerShell."

Guarda o ficheiro com a extensão .ps1 e executa com:

.\ola.ps1

⚠️ Atenção: Scripts descarregados da internet

Se descarregares um script da internet, o Windows pode marcá-lo como potencialmente perigoso. Mesmo que o conteúdo seja seguro, o PowerShell pode bloquear a execução.

Para desbloquear o ficheiro, usa:

Unblock-File -Path .\script_descarregado.ps1

Este passo é essencial se estiveres a testar scripts obtidos de blogs, fóruns ou repositórios públicos.

Mais informação Consulta a documentação oficial da Microsoft: 🔗 about_Scripts

❗Nota: Se ao tentares correr o teu script aparecer um erro a dizer que a execução de scripts está desativada, não te preocupes — vamos já ver porquê e como resolver isso.

1.5. 🧩 Exercício

Cria um script chamado apresentacao.ps1 que:

  • Guarda o teu nome numa variável
  • Usa Write-Output para mostrar uma mensagem de apresentação
  • Mostra a data atual com Get-Date

Exemplo de saída ao correr o script:

Olá, o meu nome é Nuno.
Hoje é:
domingo, 21 de abril de 2025 09:42:17

Guarda e executa com:

.\apresentacao.ps1
Resposta 
# apresentacao.ps1
# Apresentação simples com nome e data atual

$nome = "Nuno"
Write-Output "Olá, o meu nome é $nome."
Write-Output "Hoje é:"
Get-Date

1.6. Boas práticas iniciais com scripts

A partir do momento em que começas a escrever scripts, é importante adotar boas práticas simples que vão facilitar o teu trabalho no futuro:

  • Guarda os scripts em pastas organizadas
    Cria uma pasta como MeusScripts e guarda lá os teus ficheiros .ps1 para os localizares facilmente.

  • Executa scripts a partir da pasta onde estão guardados
    Usa cd para navegar até à pasta onde o script está e depois executa com:

    .\nome_do_script.ps1

  • Evita erros com caminhos absolutos e relativos
    Se o teu script usa ficheiros (por exemplo, lê ou escreve em .txt), um caminho como este:

    .\dados\ficheiro.txt

    só vai funcionar se correres o script a partir da mesma pasta onde ele está.

    Se correres o script a partir de outra pasta, o caminho relativo pode falhar porque está a ser interpretado em relação à pasta atual do terminal, não à localização do script.

    ➡️ Boa prática: dentro do script, usa $PSScriptRoot para construir caminhos relativos à localização do próprio script, por exemplo:

    $ficheiro = Join-Path $PSScriptRoot "dados\ficheiro.txt"

    Isto garante que o script funciona corretamente mesmo que seja chamado a partir de outras localizações.

  • Começa sempre com comentários descritivos
    Inclui uma ou duas linhas no início do script a explicar o que ele faz:

    # Este script mostra o nome do utilizador e a data atual

  • Separa a lógica por blocos
    Usa linhas em branco e comentários para organizar a estrutura:

    # Secção 1 – Definir variáveis
# Secção 2 – Mostrar mensagens

1.7. 🧩 Exercício – Corrigir caminhos relativos com $PSScriptRoot

Vamos simular um pequeno projeto com ficheiros organizados em pastas e perceber porque é que os caminhos relativos podem causar erros.

Passo 1 – Preparar a estrutura

  1. Cria uma pasta chamada Projeto1
  2. Dentro dessa pasta, cria uma subpasta chamada dados

  3. Dentro da pasta dados, cria um ficheiro chamado mensagem.txt com o conteúdo:

    Bem-vindo ao PowerShell!

Passo 2 – Criar um script com caminho relativo

  1. Na pasta Projeto1, cria um script chamado ler.ps1 com o seguinte conteúdo:

     # ler.ps1
 # Tentativa de ler um ficheiro usando caminho relativo (forma incorreta)

 $ficheiro = ".\dados\mensagem.txt"
 Get-Content $ficheiro

Passo 3 – Executar o script fora da pasta

  1. Abre o PowerShell, navega até a localização do script e corre o mesmo .\ler.ps1

  2. Agora navega até outra pasta qualquer (por exemplo, C:\Utilizadores\TeuNome).

  3. Executa o script com o caminho completo:

     C:\Caminho\Para\Projeto1\ler.ps1

    Ou:

     cd C:\Caminho\Para\
 .\Projeto1\ler.ps1

  4. Observa o erro que aparece.
    O script não consegue encontrar o ficheiro porque o caminho . \dados\mensagem.txt é relativo à localização atual do terminal, e não à localização do script.

O teu desafio

Corrige o script para que funcione corretamente, independentemente da pasta onde o terminal está aberto.

  • Usa a variável automática $PSScriptRoot
  • Usa Join-Path para construir o caminho completo

💡 Dica: $PSScriptRoot representa a pasta onde o script está guardado.

Resposta 
# ler.ps1
# Caminho construído corretamente com base na localização do script

$ficheiro = Join-Path $PSScriptRoot "dados\mensagem.txt"
Get-Content $ficheiro

2. Execution Policies no PowerShell

O PowerShell inclui um sistema de segurança chamado Execution Policy (Política de Execução) que define se e como scripts podem ser executados no teu sistema.

Estas políticas não são um mecanismo de segurança forte, mas servem para evitar a execução acidental de scripts maliciosos, especialmente em ambientes corporativos.

2.1. Tipos principais de Execution Policy

Política Descrição
Restricted Padrão em muitos sistemas. Não permite executar scripts. Apenas comandos interativos.
AllSigned Permite apenas scripts assinados digitalmente por uma entidade de confiança.
RemoteSigned Scripts locais podem correr. Scripts da internet precisam de estar assinados. Ideal para desenvolvimento local.
Unrestricted Permite todos os scripts, mas mostra avisos se o ficheiro vier da internet.
Bypass Nenhuma restrição. Útil para automações específicas (evitar em uso geral).

Ver a política atual:

Get-ExecutionPolicy

Ver todas as políticas por nível (MachinePolicy, UserPolicy, etc.):

Get-ExecutionPolicy -List

2.2. Alterar política

Alterar a política só para o teu utilizador:

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

⚠️ Atenção: Só deves alterar a Execution Policy se souberes o que estás a fazer. Usa sempre o -Scope CurrentUser para não afetar outros utilizadores do sistema.

Repor para o valor padrão (Restrito):

Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy Restricted

Como saber se um script veio da internet?

Se fizeres download de um .ps1 da internet, o Windows pode marcá-lo com um bloqueio. Podes remover esse bloqueio com:

Unblock-File -Path .\script.ps1
