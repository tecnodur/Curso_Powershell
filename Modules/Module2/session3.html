🧭 Sessão 3: Tratamento de Erros com try, catch e finally


🎯 Objetivo
Aprender a capturar e tratar erros em PowerShell para tornar os scripts mais robustos, previsíveis e fáceis de manter.

1. Porquê tratar erros

1.1. Robustez e previsibilidade

  • Evitar falhas inesperadas que interrompem o fluxo do script.
  • Permitir fallbacks ou caminhos alternativos (e-mails de alerta, logs, repetição de operação).
  • Melhorar a experiência do utilizador com mensagens claras.

1.2. Segurança e manutenção

  • Prevenir exposição de dados sensíveis em mensagens de erro cruas.
  • Facilitar o debug e a manutenção futura ao produzir logs detalhados.

2. Sintaxe de try / catch / finally 

try {
    <# bloco que pode lançar erro #>
}
catch [<TipoDeExceção>] {
    <# tratamento #>
}
finally {
    <# bloco que corre SEMPRE #>
}
  • try – envolve código que pode gerar erros.
  • catch – executado quando ocorre um erro terminante. Para capturar erros não-terminantes use -ErrorAction Stop.
  • finally – executado quer exista ou não erro (fechar ficheiros, remover temp, etc.).

2.1. Diferença entre erros não-terminantes e terminantes

  • Erros não-terminantes
    São erros que não interrompem a execução do script.
    O PowerShell apresenta a mensagem de erro, mas o script continua a correr.

    Exemplo:

      Get-ChildItem -Path "C:\inexistente"

    ➔ Gera um erro, mas o script continua a executar o que vem a seguir.

  • Erros terminantes
    São erros que interrompem imediatamente a execução.
    Apenas erros terminantes podem ser capturados pelo bloco catch.

    Para tornar um erro não-terminante em terminante, usa-se -ErrorAction Stop.

    Exemplo:

      Get-ChildItem -Path "C:\inexistente" -ErrorAction Stop

    ➔ Agora o erro é tratado como terminante e pode ser capturado no catch.

2.2. Múltiplos blocos catch 

try {
    # ...
}
catch [System.IO.IOException] {
    "Problema de E/S: $_"
}
catch {
    "Erro genérico: $_"
}

3. Trabalhar com diferentes tipos de exceções

  • Filtrar exceções específicas para respostas específicas.
  • Hierarquia – exceções mais específicas devem vir antes das genéricas.
  • Utilizar o alias [Exception] para a última catch genérica.
Tipo de exceção Quando ocorre Estratégia típica
System.IO.FileNotFoundException Ficheiro não encontrado Criar/solicitar ficheiro, abortar tarefa
System.UnauthorizedAccessException Permissões insuficientes Elevar privilégios, informar utilizador
Exception Qualquer outro erro Log genérico + rethrow ou saída

Como descobrir que exceções um comando pode lançar

PowerShell nem sempre documenta todas as exceções de forma explícita, mas existem várias formas práticas de descobrir:

  • Consultar a ajuda do comando
    Usa Get-Help NomeDoComando -Full para ver detalhes.
    Alguns cmdlets indicam exceções na documentação.

  • Forçar erros e inspecionar
    Executa o comando de propósito com parâmetros errados e captura o erro para ver o tipo de exceção.

    Exemplo:

      try {
      Get-Content -Path "C:\ficheiro_inexistente.txt" -ErrorAction Stop
  }
  catch {
      $_.Exception.GetType().FullName
  }

    Resultado típico:

      System.Management.Automation.ItemNotFoundException

  • Usar a variável $Error

4. A variável automática $Error

Se não capturaste o erro num catch, podes sempre inspecionar o erro mais recente guardado em $Error[0].

  • Array circular com histórico dos erros da sessão.
  • $Error[0] contém o erro mais recente. Usa .Exception para o objeto completo.

  • $Error[0].Exception.GetType().FullName

    Exemplo:

      try { 1/0 } catch {}
  $ultimo = $Error[0]
  $ultimo.Exception.GetType().FullName   # System.Management.Automation.RuntimeException
  • Limpar histórico: Clear-Variable -Name Error ou $Error.Clear().

5. 🧩 Exercício – Criar um script com tratamento de erro

Desafio: Desenvolver um script interativo que leia um caminho de ficheiro do utilizador, tente abri-lo, e trate 3 cenários de erro.

5.1. Requisitos

  1. Solicitar caminho de ficheiro via Read-Host.
  2. Usar try/catch para:
    • FileNotFoundException – perguntar se quer criar o ficheiro.
    • UnauthorizedAccessException – aconselhar execução elevada ou outro local.
    • Qualquer outro erro – registar em log erros.log.
  3. Garantir que, no bloco finally, se escreva “Operação concluída.”.

5.2. Dicas

  • Lembra-te de -ErrorAction Stop em cmdlets como Get-Content.
  • Utiliza Start-Transcript para gravar logs detalhados.
  • Para debug, experimente lançar erros manualmente: throw "Erro de teste".

5.3. Critérios de sucesso

  • Script não termina abruptamente perante erros.
  • Mensagens amigáveis e específicas.
  • Log é criado/atualizado conforme necessário.
Resposta 
$caminho = Read-Host "Introduz o caminho do ficheiro"

    try {
    Get-Content -Path $caminho -ErrorAction Stop
    Write-Output "Ficheiro lido com sucesso."
}
catch [System.Management.Automation.ItemNotFoundException] {
    Write-Output "Ficheiro não encontrado. "
}
catch [System.UnauthorizedAccessException] {
    Write-Output "Permissões insuficientes. Tenta correr como administrador ou escolhe outro local."
}
catch {
    $erro = $_.Exception.Message
    Add-Content -Path "erros.log" -Value "Erro: $erro"
    Write-Output "Erro inesperado registado em erros.log"
}
finally {
    Write-Output "Operação concluída."
}
